# 目录说明

这个文件用于解释工作区为什么分成 `.github/`、`组合/`、`项目/`、`共享/` 四类区域，以及每类区域应该放什么。

## 总体原则

- `.github/` 放“工具识别型配置”。
- `组合/`、`项目/`、`共享/` 放“项目运行中的事实和记忆”。
- 根目录只保留总入口文件和高层说明，不堆具体项目状态。

## 顶层目录职责

| 目录或文件              | 职责             | 典型内容                                 |
| ----------------------- | ---------------- | ---------------------------------------- |
| `.github/`              | Agent 配置层     | custom agents、instructions、工作区说明  |
| `AGENTS.md`             | 工作区总规则入口 | 全局运行模型、目录约定、角色边界         |
| `组合/`                 | 跨项目状态层     | 组合索引、路线图、依赖关系、周报         |
| `项目/`                 | 单项目状态层     | 项目简报、项目状态、交接、验证、环境说明 |
| `共享/模板/`            | 复用模板层       | 任务包模板、交接模板、状态模板           |
| `多Agent交付架构_V1.md` | 架构设计文档     | 角色模型、分层设计、实施路线             |
| `Pilot快速开始.md`      | 启动说明         | 如何先跑通最小 pilot 流程                |

## 为什么 `.github/` 要单独放

`.github/` 里的文件主要是给 VS Code 和 Copilot 识别的，不是给项目状态本身用的。

例如：

- `.github/copilot-instructions.md` 会被当成工作区级说明自动加载。
- `.github/agents/` 里的 `.agent.md` 文件会出现在 Agent 角色里。
- `.github/instructions/` 里的 `.instructions.md` 会按匹配规则自动生效。

如果把这些文件混到 `项目/` 或 `组合/` 里，结构会更乱，而且不利于后续扩展和排查。

## 为什么项目状态不放进 `.github/`

因为项目状态是持续变化的数据，不是工具配置。

例如下面这些内容都应该放在 `项目/` 下：

- 当前目标是什么
- 下一个任务是什么
- 刚刚改了哪些文件
- 哪些测试跑过了
- 当前环境缺什么

这些内容是 Agent 切换后要恢复上下文时读取的“事实”，而不是要求工具始终自动加载的规则。

## 推荐使用方式

1. 需要定义 Agent 行为或规则时，优先看 `.github/`。
2. 需要看项目做到哪一步时，优先看 `项目/`。
3. 需要看跨项目优先级时，优先看 `组合/`。
4. 需要复用文档骨架时，优先看 `共享/模板/`。

## 结构整理建议

当前结构已经适合 pilot 使用，后续建议继续保持这几个原则：

- 不把项目状态文件塞进 `.github/`。
- 不把 Agent 定义散落到项目目录里。
- 不把模板和真实项目文件混放。
- 根目录只保留总入口、说明和架构文件。
