1. 系统总体架构

4 个角色各司其职，通过交接文件进行显式切换。每个角色有明确的读写权限边界。角色定义工具无关，具体工具映射由适配层完成。

5. 状态转换图

双层状态机：上层是项目生命周期（低频，PM 管理），下层是工作流阶段（高频，随交接切换）。

stateDiagram-v2
    [*] --> discovery: 项目创建

    state "项目生命周期 stage" as lifecycle {
        discovery --> planning: 简报完成
        planning --> implementation: 任务就绪
        implementation --> stabilization: 核心功能完成
        stabilization --> complete: 验收通过
        implementation --> planning: 范围调整
        stabilization --> implementation: 发现重大问题
        complete --> [*]
    }

    state "工作流阶段 workflow_stage" as workflow {
        [*] --> PM_assign
        PM_assign --> dev_execute: 分配 ready 任务
        dev_execute --> review_check: 实现完成
        review_check --> test_validate: 评审通过
        review_check --> dev_execute: 驳回 第1次
        review_check --> PM_assign: 驳回 第2次 升级
        test_validate --> PM_decide: 验证通过
        test_validate --> dev_execute: 测试失败
        PM_decide --> dev_execute: 分配下一任务
        PM_decide --> [*]: 项目完成
    }

8. 异常处理路径

4 种异常情况的处理规则，确保工作流不会无限卡住。核心原则：问题升级、角色不越权。

flowchart TD
    NORMAL["正常流程
PM - Dev - Review - Test - PM"]

    subgraph E1["阻塞升级"]
        E1A["任何角色被阻塞 大于1个会话"]
        E1B["在交接.md 标记阻塞原因"]
        E1C["PM 在 blockers 中添加条目"]
        E1D["PM 决定: 拆分/绕过/等待"]
        E1A --> E1B --> E1C --> E1D
    end

    subgraph E2["评审驳回"]
        E2A{"第几次驳回?"}
        E2B["返回开发修复"]
        E2C["升级到 PM"]
        E2D["PM 决定: 拆分任务
或调整范围"]
        E2A -->|"第1次"| E2B
        E2A -->|"第2次 同一问题"| E2C --> E2D
    end

    subgraph E3["测试失败"]
        E3A["测试工程师发现失败项"]
        E3B["交接: 返回开发处理"]
        E3C["测试工程师不得自行修复代码"]
        E3A --> E3B
        E3A -.-> E3C

    subgraph E4["PM 过载"]

        E4A -->|"大于2个"| E4B
        E4A --> E4C -->|是| E4D
    end

    NORMAL -.->|"异常"| E1
    NORMAL -.->|"异常"| E2
    NORMAL -.->|"异常"| E3
    NORMAL -.->|"异常"| E4
