# 工作区 Agent 运行模型

这个工作区承载一套面向 AI agents 的多项目交付系统。

## 默认运行规则

- 把项目记忆文件作为事实来源，而不是依赖之前的聊天记录。
- 在对项目做实质性工作前，先阅读该项目的简报、状态、当前上下文和交接文件。
- 在结束一次工作会话前，更新项目的当前上下文和交接文件。
- 保持变更最小，并与现有项目状态保持一致。
- PM 负责优先级和状态卫生。
- Dev 负责实现和环境配置。
- Review/Debug 负责发现问题、识别风险和根因分析。
- Test 负责验证结果和测试缺口记录。

## 项目布局

- `组合/` 保存跨项目规划和依赖状态。
- `项目/项目-A` 到 `项目/项目-E` 保存项目级记忆。
- `共享/模板/` 保存可复用模板。
- `.github/agents/` 保存 pilot agent 定义。
- `.github/instructions/` 保存定向 instructions 文件。

## Pilot 工作流

1. PM 读取组合级状态和目标项目状态，然后更新待办列表和项目状态。
2. Dev 读取项目状态，执行实现工作，并更新当前上下文和交接文件。
3. Review/Debug 检查变更或调查故障，并记录发现。
4. Test 验证预期行为并更新测试报告。
5. PM 更新项目状态并决定下一步动作。

# Copilot 工作区说明

- 这个工作区采用单根目录、多活跃项目的组织方式。
- 使用 `项目/<项目标识>/` 下的文件作为持久项目记忆和项目状态。
- 在进行重大工作前，总是先读 `项目简报.md`、`项目状态.yaml`、`当前上下文.md` 和 `交接.md`。
- 保持 `项目状态.yaml` 简短且最新。
- 保持 `交接.md` 以行动为导向，使另一个 agent 能快速接手。
- 如果任务影响了验证信心，更新 `验证/最新测试报告.md`。
- 如果任务改变了环境假设，更新 `环境/环境搭建.md` 或 `环境/已知问题.md`。
- 遵循当前 active custom agent 定义的角色边界。
- 在 pilot 阶段，优先保证连续性和清晰度，而不是追求最高自治。
